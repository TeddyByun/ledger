적재 플로우

업로드 → 거래 생성

각 단계가 어떤 API를 호출하고 어떤 테이블에 무엇을 쓰는지. 잡 상태는 queued → parsing → classifying → review → completed 순으로 전이한다.

1

로그인 · 가구 컨텍스트 확보

왜 먼저인가 — 모든 도메인 데이터가 householdId로 격리되므로, 요청 주체·가구가 정해져야 이후 모든 읽기·쓰기가 가능하다.

이메일·비밀번호로 로그인하면 Access 토큰(15분, 메모리)과 Refresh 토큰(30일, httpOnly 쿠키)을 받는다. 이후 모든 요청은 Authorization: Bearer … 와 활성 가구를 가리키는 X-Household-Id 를 함께 보낸다.

POST/auth/login usermembershiprefresh_token

응답: { accessToken, user, household:{ id, role } } — 이후 서버는 요청마다 { userId, householdId, role } 컨텍스트를 주입해 데이터를 자동 필터한다.

2

명세서 업로드 → 적재 잡 생성

파싱은 무거우므로 즉시 처리하지 않고 큐(BullMQ)에 넣고 jobId만 돌려준다.

발급사(예: 하나카드), 결제수단, 파일을 함께 전송한다. 서버는 파일을 스토리지에 저장하고 import_job 한 건을 만든 뒤 워커에 위임한다.

POST/imports import_job

응답: { jobId } · 화면은 GET /imports/{jobId} 를 폴링(또는 SSE)해 진행률을 본다.

queued→parsing→classifying→review→completed
3

파싱 · 정규화

카드사마다 컬럼이 다르다 — 발급사별 파서가 공통 스키마로 맞춘다.

명세서를 두 층으로 나눈다: 헤더(요약) 1건과 이용상세내역 N건. 핵심 금액 규칙은 가계부 금액 = 결제원금(principal) + 수수료(fee) — 할인이 반영된 실청구액이며, 할부는 그 달 청구 회차 금액만 잡는다.

workerparse() EUC-KR → UTF-8

예: 대성석유 주유소 · 이용 74,000 · 할인 −2,580 → 결제원금 71,420 (일시불이라 수수료 0).

4

원천 적재 (staging)

먼저 무손실로 저장한다 — 아직 가계부 거래가 아니라 "원본 보관"이다.

헤더는 card_statement 에, 건별 내역은 card_transaction 에 들어간다. 이 시점엔 transaction_id = NULL, is_classified = 'N'.

card_transaction — 적재 직후 (분류 전)
txn_datemerchant_nameusageprincipalfeeclassified
2026-03-01대성석유(주)…주유소74,00071,4200N
2026-03-16365정형외과의원100,110100,1100N
5

자동분류 + 대사(reconciliation)

여기서 원천이 가계부 거래가 된다 — 규칙으로 분류하고, 중복 지출을 걷어낸다.

자동분류: 가맹점명을 merchant_category_map 패턴에 맞춰 분류코드를 부여한다(예: "주유소" → 09 차량). 매칭되면 transaction 을 만들고 원천에 transaction_id 로 연결(is_classified='Y').

미매칭은 pending 으로 남아 잡이 review 상태가 되고, 사용자가 검토 화면에서 확정한다.

대사: 카드대금이 은행에서 빠져나간 출금(타사카드 결제)은 카드 건별 지출과 중복이므로 거래로 만들지 않고 card_settlement 로 제외한다. 본인 계좌 간 이체는 self_transfer.

PATCH/imports/{jobId}/classify transactionmerchant_category_map
6

월 집계 재계산 → 완료

대시보드를 빠르게 하려고, 조회 때 합산하지 않고 미리 합산해 저장한다.

영향받은 달(예: 2026-03)의 monthly_summary·monthly_category_stat·monthly_payment_stat 를 삭제 후 재삽입(rebuild)한다. 잡이 completed 가 되면 이제 조회 가능.

집계 대상 규칙 — amount IS NOT NULL AND status='settled' 인 거래만 합산된다. 정보성 행(status='info')·자기이체·카드대금은 애초에 transaction에 연결되지 않아 자동으로 빠진다.

조회는 어떻게 되는가

두 개의 경로

화면이 데이터를 읽는 방식은 목적에 따라 둘로 갈린다. 하나는 개별 거래를 직접 훑고, 다른 하나는 5단계에서 미리 만들어 둔 집계를 즉시 읽는다.

경로 A · 거래 목록

"3월에 쓴 내역을 보여줘" — 상세·필터·검색·페이지네이션

transaction 을 직접 조회하고 분류·결제수단·상대를 조인한다. 대량이므로 커서(keyset) 페이지네이션으로 무한스크롤한다.

GET/transactions

?from & to & categoryCode & paymentMethodId & cursor

경로 B · 대시보드/통계

"이번 달 분류별 비중" — 즉시 뜨는 요약·차트

거래를 매번 합산하지 않고 monthly_category_stat 등 집계 테이블을 그대로 읽는다. 그래서 데이터가 많아도 조회가 빠르다.

GET/stats/monthly

?ym=2026-03 → 도넛·추세 차트 데이터

데이터 계보

원천 → 가계부 → 집계

한 건의 카드 이용이 어느 테이블을 거쳐 어디로 조회되는지. 왼쪽은 무손실 원천, 가운데는 분류된 가계부 거래, 오른쪽은 대시보드용 집계다.

원천 / Staging

card_statement

  • 명세서 헤더(월·합계·결제일)
  • 카드·월별 1건
│ 1 : N
원천 / Staging

card_transaction

  • 건별 이용내역(무손실)
  • principal · fee · 할부
─ 자동분류 → 생성·연결 ─
가계부 / Ledger

transaction

  • amount = principal + fee
  • category_code (분류)
  • status = settled / pending / info
  • ← 경로 A 가 읽는 곳
↑ category · payment_method 조인
─ 월 rebuild ─
집계 / Aggregate

monthly_category_stat

  • ym · category_code
  • amount_total · ratio
  • ← 경로 B 가 읽는 곳
＋ monthly_summary · payment_stat
수입 (income) 지출 (expense) 집계 제외 (self_transfer · card_settlement · info) 강조 / 링크

경로 A · B 예시

개념이 아니라 실제에 가깝게. 아래는 3월 지출을 조회하는 두 방식과 그 결과다. (가구 스코프 household_id = 1 는 서버가 자동 주입한다.)

경로 A — 거래 목록 (transaction 직접 조회)

-- GET /transactions?type=expense&from=2026-03-01&to=2026-03-31
SELECT t.transaction_date, c.name AS category, t.description,
       t.amount, p.name AS payment, t.status
FROM "transaction" t
JOIN category c        ON t.category_code = c.code
JOIN payment_method p  ON t.payment_method_id = p.id
WHERE t.household_id = 1
  AND t.type = 'expense'
  AND t.status = 'settled'
  AND t.transaction_date BETWEEN '2026-03-01' AND '2026-03-31'
ORDER BY t.transaction_date DESC, t.id DESC   -- 커서 페이지네이션 키
LIMIT 50;

SELECT category_code, amount_total, ratio
FROM monthly_category_stat
WHERE ym = '2026-03' AND household_id = 1
ORDER BY amount_total DESC;
